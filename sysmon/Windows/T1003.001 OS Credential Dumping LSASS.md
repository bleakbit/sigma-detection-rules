##### Hunting in Security Onion

Sysmon EID 10: To hunt on potential lsass.exe data dumping, a threat hunter can look for any process that has been given any access to lsass.exe. While the above rule alerts on certain access (0x1fffff for full access and 0x1010 for process vm read), attackers may use tools to that will throw off a different granted access code. By filtering on winlog.event_data.TargetImage = lsass.exe and looking at the winlog.event_data.GrantedAccess field, analysts can filter through the process.executable field to see all processes that were given any access to lsass.exe. From there, analysts can look at what accesses have been granted to each process to investigate any potential abnormal activity.
```
winlog.event_data.TargetImage: *lsass.exe AND winlog.event_data.GrantedAccess: *
```

Furthermore, if this can be hunted for using Windows Logs event code 4663. An example query looks like this: event.code: "4663" AND winlog.event_data.ObjectName:_lsass_. This will entail sifting through all processes that have requested access to lsass, not just malicious use cases.
```
event.code: "4663" AND winlog.event_data.ObjectName:*lsass
```

##### Hunting in Splunk

index="sysmon" TargetImage=*lsass* GrantedAccess IN (0x1010, 0x1fffff) | table _time, TargetImage, process_exec, SourceImage, GrantedAccess
